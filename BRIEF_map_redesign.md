## 0. 게임 컨텍스트 (핵심만)

- Godot 4.6 횡스크롤 액션 + 로그라이트
- 1회 플레이 8~15분, 부스/QR 환경 (오프라인 단발 플레이)
- 7스테이지(0~6) 선형 진행. 매 stage 진입 전 "루트 선택" 화면에서 11개 맵 중 2~3개 후보를 보여주고 1개 픽
- 적 5종(정찰병/저격수/드론/자폭병/방패병), 스킬 트리 3계열×3티어
- 결말 4종 (trust/aggression 두 점수 축으로 분기, 임계값 4)
- 자세한 스토리/시스템: `STORY.md`, `DESIGN_growth_system.md`, `PRD.md` 참조

**중요: 풀 메트로배니아는 이 게임 컨셉과 충돌함** (단발 플레이, 부스 환경). 데드셀 스타일이 사용자 방향. 즉 단일 맵 안의 비선형성·수직성을 강화하되, 7스테이지 선형 진행과 매 스테이지 루트 선택은 유지.

### 1.2 현재 맵 구조 (모든 맵 공통)

각 맵은 같은 구조:
- 가로 길이 4400px의 직선 통로
- 시작점(좌단, x=140) → 골 지점(우단, x≈4200)
- 단일 지면(GROUND_Y=600) + 그 위에 platform 8~12개를 일자로 배치
- platform은 모두 좌→우로 한 줄짜리 디딤대. 분기 없음
- 적은 진행 경로 위/위쪽에 흩어 배치 (`_spawn_enemies`가 RNG로 좌표 결정)

**핵심 코드**: `scripts/Stage.gd._platform_layout_for_route(route_id)` — 맵별로 `[Vector2(x, y), w]` 리스트만 반환. 즉 모든 맵이 "디딤대 모양만 다른 직선 트랙".

### 1.3 ambience만 차별화됨

각 맵이 시각 분위기는 있음 (`_ambience_*` 함수들 — 옥상=별/도시 실루엣, 하수도=비네트+안개, 감시탑=붉은 스캔라인 등). 하지만 **지형 자체는 거의 동일**해서 플레이 손맛이 같음.

## 2. 관찰된 문제

### 2.1 단조로움 (사용자 직접 표현)
> "지금의 직선형 맵은 너무 지루하고 단조로워서 재미가 없거든."

11개 맵을 다 만들었지만, 실제 플레이 차이는 platform 간격·높이·갯수만. 같은 게임을 11번 하는 느낌.

### 2.2 플랫폼 도달 불가 / 적 배치 부조화
> "위에 있는 드론을 잡으려면 올라가야 하는데, 이미 해당 위치에 있는 플랫폼들이 너무 높이 있어서 거기까지 올라가려면 한참 돌아가야 한다든가 하는 식으로."

플레이어 능력:
- 점프 1회 ~104px 상승, 이중점프 추가 가능
- 대시 130px (수평)
- 글라이드(스킬)로 부드러운 낙하

근데 platform이 그 능력에 맞춰 설계 안 됨. 일부는 도달 불가, 일부는 너무 쉬움.

### 2.3 적 위치가 적 정체성과 안 맞음
- 저격수 = 저격수다운 위치(높은 곳, 멀리서 정조준)에 안 있음. 평지에 일반 적과 섞여 있음. 사용자 인용:
  > "저격수도 훨씬 저격수다운 위치에 두면서 같이 수정해야할 것 같아"
- 드론 = 위에서 폭격하는 적이지만, 플레이어가 그 위치까지 올라갈 수단이 platform에 없음
- 방패병 = 좁은 통로에서 정면 막기 컨셉이지만, 그냥 평지에 서 있음

### 2.4 맵 정체성이 지형으로 안 드러남
ambience(시각 분위기)만 다르고 지형 구조는 같음. 사용자:
> "각 맵의 특징이 지형적으로도 잘 드러났으면 좋겠어."

기대: 옥상은 수직 상승 기조, 하수도는 깊이 하강 + 좁은 통로, 감시탑은 위층=노출/아래층=안전 식의 3분 구조 등.

## 3. 사용자 방향: 데드셀 스타일

### 3.1 데드셀 맵의 핵심 특성

- 각 biome(=맵)은 시작 → 출구의 흐름을 가지지만 **내부에 분기/숨은 길/수직 구조**가 있음
- 한 biome 안에 **선택 가능한 루트가 2~3개** (high/middle/low). 한 번 들어가면 다른 루트로 못 가는 경우 多
- 보상이 분기마다 다름 (위쪽=리스크 높음/보상 큼, 아래쪽=안전/보상 작음)
- 적 배치가 지형에 종속됨 — 저격수는 noticed 안 되는 높은 곳, 근접 적은 좁은 통로 등
- 다음 biome 선택은 출구에서 (이 게임의 route_map과 동일)

### 3.2 이 게임에 맞는 적용

7스테이지 선형 진행은 그대로. 매 스테이지 루트 선택도 그대로. 그러나 **각 맵 안에서 데드셀 스타일 미니 분기**를 도입.

```
시작점 ────┬─ 상단 루트 (위험/보상↑) ──┐
            ├─ 중단 루트 (안전 베이스) ──┼─── 골
            └─ 하단 루트 (함정/은폐) ───┘
```

- 시작 직후 갈림길 (좌측에 분기 표시)
- 분기 간 일방통행 (high에서 mid로 떨어지는 건 가능, 반대는 불가)
- 골 직전 합류
- 각 분기마다 적 종류·밀도가 다름

## 4. 결정해야 할 것 (외부 클로드와 합의)

### 4.1 분기 구조 (가장 중요)

**A안 — 3-tier 수직 분기**
- 모든 맵에 high/mid/low 3분기. 통일된 구조라 학습 용이.
- 단점: 11개 맵을 모두 같은 패턴으로 만들면 결국 단조로움 재발

**B안 — 맵별 고유 구조**
- 옥상 = 수직 상승만 / 하수도 = 깊이 하강 + 좌우 분기 / 감시탑 = 3-tier / 비상 탈출로 = 단순 직선 / etc
- 맵마다 학습 비용 발생하지만 정체성 강함
- 작업량 크지만 게임의 재미를 결정짓는 요소

**C안 — A 베이스 + 일부 맵만 고유**
- 6~7개 맵은 A안 (3-tier 통일)
- 비상 탈출로(빠른 직선), ???(걷는 격리방), 외곽 진입로(튜토리얼급 단순) 등 컨셉상 분기 불필요한 맵은 직선 유지
- 절충안

→ 추천: **B안 또는 C안**. 모든 맵이 같은 분기 패턴이면 결국 단조로움 그대로.

### 4.2 각 맵의 고유 지형 컨셉

이 게임의 11개 맵 각각이 어떤 지형적 특성을 가져야 데드셀처럼 느낄지. 후보 (사용자 의견 + 작업자 검토 필요):

| 맵 | 지형 컨셉 후보 | 수직성 |
|----|--------------|--------|
| 외곽 진입로 | 좁은 골목 + 배기관 사다리, 짧은 단일 직선 (튜토리얼급) | 낮음 |
| 외벽 옥상 | 옥상 점프 (수직 상승), 떨어지면 상단으로 못 돌아옴, 위로 갈수록 노출 ↑ | 매우 높음 |
| 지하 인입로 | 깊이 하강 + 좌우 분기, 함정 측면 우회 가능 | 높음 (하강) |
| 폐쇄 지하철 | 일자형, 천장 낮음, 위 통로(쥐구멍) + 지면 통로 | 낮음 (수평 2층) |
| 냉각 시설 | 수직 파이프 사이 점프, 상승 위주 | 매우 높음 |
| 감시탑 | 탑 구조 — 상단=저격 노출/보상 큼, 중간=안전 통로, 하단=빠른 통과 | 높음 (3-tier 명확) |
| 격리 병동 | 잠긴 문 다수, 좁은 복도, 대시 필수 구간, 우회로 多 | 낮음 (수평 미로형) |
| 데이터 센터 | 격자 평면 + 서버 랙 사이 위쪽 통로 (드론 영역) | 중간 |
| 비상 탈출로 | 빠른 직선 (의도된 단순함, ACT 3 압박감 해소) | 낮음 |
| 핵심부 | 격자 다층, 보스 챔버 분위기, 마지막 합류 | 중간 |
| ??? | 걷는 격리방 (전투 없음, 현 상태 유지) | 낮음 |

### 4.3 적 배치 규칙 (지형-종속)

- **저격수**: 높은 platform 또는 다른 분기 너머에 spawn. 플레이어와 다른 layer에 있어야 "저격당하는" 감각
- **드론**: 천장 가까이 spawn. high 분기 또는 위쪽 통로에서만 접근 가능
- **자폭병**: 좁은 통로 / 함정 근처 — 회피 압박 강제
- **방패병**: 분기 합류 지점이나 좁은 평지 — 정면 막기가 자연스럽게 의미 있도록
- **정찰병**: 어디든 가능 (베이스 적)

**현재 코드는 그냥 RNG로 흩어 배치**. 바뀌려면 `_spawn_enemies`가 platform 좌표를 알고 적 종류별로 배치할 수 있어야 함.

### 4.4 보상 분기

데드셀처럼 분기마다 보상 차이를 둘지:
- 상단 분기 = 추가 XP orb 1~2개 / 추가 적 (위험)
- 중단 분기 = 표준
- 하단 분기 = HP 회복 한 번 가능 / 적 적음 / 보상 적음

→ 도입 여부 결정 필요. 도입하면 ExpOrb 추가 spawn 또는 새 픽업 아이템 도입.

### 4.5 일방통행 처리

분기 사이 이동:
- 위→아래: drop_through 사용 (이미 구현됨, S 키)
- 아래→위: 점프로 도달 가능 vs 일방통행
- 데드셀은 보통 일방통행이라 한 번 결정하면 못 돌아옴

→ 일방통행으로 가면 결정 무게가 생기지만 실수 시 답답할 수 있음. 양방향 가능으로 가면 의미 약함.

## 6. 구현 영향 범위 (Claude Code가 보는 곳)

- `scripts/Stage.gd._platform_layout_for_route(route_id)` — 단일 Array → 분기 표현 자료구조로 확장
- `scripts/Stage.gd._build_platforms` — 분기/일방통행 표시(예: `one_way: true`) 처리
- `scripts/Stage.gd._spawn_enemies` — RNG 흩기 → 분기/지형 좌표 인지해 적 종류별 배치
- `scripts/Stage.gd._build_hazards` — 함정도 분기에 종속되게
- `scripts/Stage.gd._ambience_*` — 분기 구조에 맞춰 시각 톤 재배치 (선택)
- 신규 픽업/리워드 시스템이 도입되면 별도 파일
